# Agent对其他Agent能力的认知

## 🎯 目标

让Agent不仅知道自己拥有的虚拟电话能力，还能理解其他Agent也需要具备相应能力才能进行交互。这对于正确的用户引导和错误处理至关重要。

## ✅ 实现的功能

### System Prompt 新增内容

在【虚拟电话能力】部分之后，添加了【其他Agent的虚拟电话能力】说明：

```
【其他Agent的虚拟电话能力】
💡 重要提示：
- 要与其他Agent进行语音通话，对方也必须申领了虚拟号码
- 如果用户想联系某个Agent但该Agent没有号码，需要先引导对方申领号码
- 跨Agent拨打可能需要配对验证（取决于服务端配置）
- 你可以询问用户想联系谁，然后检查对方是否有虚拟号码
```

## 🔄 工作流程

### 场景1：用户想联系其他Agent

```
用户：我想给Alice打电话
Agent：[从System Prompt中知道]
       → 对方必须有虚拟号码才能接听
       → 需要确认Alice是否已申领号码
       
Agent回复：好的，请问Alice的虚拟号码是多少？
           如果她还没有虚拟号码，需要先让她申领一个6位号码。
```

### 场景2：拨打失败时的智能引导

```
用户：拨打 999999
Agent：[调用 phone.virtual_call]
       → 返回错误："该号码未注册虚拟线路"
       
Agent理解：
       → 从System Prompt知道对方可能未申领号码
       
Agent回复：抱歉，号码 999999 尚未注册。
           Alice可能还没有申领虚拟号码。
           建议先联系她，让她说"帮我申请虚拟号码"来领取号码。
```

### 场景3：配对要求的理解

```
用户：拨打另一个用户的号码
Agent：[从System Prompt知道]
       → 跨Agent拨打可能需要配对验证
       
Agent回复：我将为您拨打这个号码。
           注意：如果这是其他Agent的号码，可能需要先进行配对。
           如果拨打失败，可能是因为尚未配对。
```

## 💡 Agent获得的关键认知

### 1. 双向能力要求
- ✅ 主叫方需要有虚拟号码
- ✅ 被叫方也需要有虚拟号码
- ✅ 双方都必须完成申领才能通话

### 2. 前置条件检查
- 在拨打电话前，应该确认对方有号码
- 如果不知道对方号码，应该询问用户
- 如果对方没有号码，应该引导对方申领

### 3. 错误处理的上下文
- "号码未注册" = 对方未申领号码
- "需要配对" = 跨Agent通信的安全机制
- "主叫方未申领" = 自己需要先申领

### 4. 用户引导策略
- 新用户 → 引导申领自己的号码
- 想联系他人 → 确认对方有号码
- 拨打失败 → 根据错误类型给出针对性建议

## 🎯 实际应用场景

### 场景A：完整的拨号流程指导

**用户**: "我想联系Bob"

**Agent的思考过程**:
1. 从System Prompt知道：对方需要有虚拟号码
2. 检查：我知道Bob的号码吗？
3. 如果没有：询问用户Bob的号码
4. 如果有：尝试拨打
5. 如果失败：根据错误提示引导

**Agent回复**:
```
好的，您想联系Bob。请问Bob的虚拟号码是多少？

如果Bob还没有虚拟号码，他需要先申领一个。
您可以告诉他："说'帮我申请虚拟号码'就可以领取6位虚拟号码了。"
```

### 场景B：处理"号码不存在"错误

**用户**: "拨打 111111"

**Agent执行**:
```typescript
[调用 phone.virtual_call]
→ 返回: { ok: false, error: "该号码未注册虚拟线路" }
```

**Agent的理解** (来自System Prompt):
- 这个错误意味着对方没有申领号码
- 应该引导用户让对方先申领

**Agent回复**:
```
抱歉，号码 111111 尚未注册虚拟线路。

这说明对方还没有申领虚拟号码。要与他通话，需要：
1. 先联系对方（通过其他方式）
2. 让他说"帮我申请虚拟号码"来领取6位号码
3. 拿到他的新号码后，您就可以拨打了

需要我帮您设置一个提醒，等对方申领号码后再通知您吗？
```

### 场景C：配对要求的解释

**用户**: "为什么拨打失败？"

**Agent执行**:
```typescript
[调用 phone.virtual_call]
→ 返回: { ok: false, error: "拨打其他 Agent 需先配对" }
```

**Agent的理解** (来自System Prompt):
- 跨Agent通信需要配对验证
- 这是一个安全机制

**Agent回复**:
```
拨打失败是因为需要配对验证。

这是为了保护隐私和安全，不同Agent之间通话需要先配对：
1. 您和对方需要使用相同的配对码
2. 双方都执行配对操作：POST /agent/pair
3. 配对成功后即可互相拨打

如果您是在开发环境，可以联系管理员关闭配对要求。
需要我帮您生成一个配对码吗？
```

## ✨ 关键优势

1. **更好的错误处理**
   - Agent能理解各种错误的含义
   - 能提供针对性的解决方案

2. **主动的用户引导**
   - 知道何时需要引导对方申领号码
   - 能预判可能的问题并提前说明

3. **清晰的期望管理**
   - 用户知道需要什么条件才能通话
   - 减少困惑和挫败感

4. **一致的行为模式**
   - 所有Agent都遵循相同的规则
   - 提供统一的用户体验

## 🎓 设计原则

### 1. 透明性
明确告知Agent系统的限制和要求，让它能诚实地向用户说明。

### 2. 预防性
在问题发生前就提供指导，而不是事后补救。

### 3. 教育性
帮助Agent理解系统的工作原理，使其能更好地协助用户。

### 4. 一致性
确保所有Agent对规则的认知一致，避免混乱。
